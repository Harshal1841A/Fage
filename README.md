FAGE (Fraud Analytics & Governance Engine) is an enterprise-grade anti-money-laundering (AML) detection platform that combines high-dimensional machine learning, explainable AI (SHAP), real-time alerting, and LLM-powered SAR report generation into a single cohesive system.

---

## 🛑 The Problem
Current Anti-Money Laundering (AML) systems suffer from massive false-positive rates (often exceeding 90%). When legitimate transactions are flagged, human analysts waste countless hours investigating, costing banks billions annually. Furthermore, traditional "black-box" ML models fail to meet strict regulatory requirements for explainability.

## 💡 The Solution: FAGE
FAGE solves this by utilizing a highly optimized, multi-model ensemble trained on imbalanced data to accurately detect true fraud while minimizing false positives. Crucially, it integrates **Explainable AI (SHAP)** to provide a clear, human-readable reason for every flagged transaction, and leverages an LLM to auto-generate Suspicious Activity Reports (SAR), cutting analyst investigation time by up to 80%.

## 🧠 How We Built It & Technical Challenges
Building an enterprise-grade AML system in a hackathon timeline required overcoming several massive hurdles:
1. **Real-Time Data Streaming:** We needed the frontend to instantly reflect fraud alerts. We implemented **Server-Sent Events (SSE)** via FastAPI to push real-time alerts to the React frontend without heavy WebSockets.
2. **Model Serving Speed:** Running SHAP explainability on a 7-model soft-voting ensemble is computationally heavy. We optimized the pipeline to serve both predictions and feature-level attributions in near real-time.
3. **Imbalanced Data:** Training on highly skewed fraud datasets meant standard 0.5 thresholds failed. We engineered a dynamic cost-optimization threshold system (explained below) to maximize real-world precision and recall.

## 📊 Data Architecture & Model Performance

### Synthetic Metadata Wrapper
The feature vectors evaluated by the machine learning models are **real transaction data**. However, the surrounding transaction metadata (e.g., `sender_id`, `receiver_id`, `amount`, and `status`) are synthetic, randomly generated by `seed_real_data.py` to make the data look like real banking transactions in the dashboard. These are synthetic accounts scored against real transaction feature data.

### Resolving Thresholds (0.5 vs Optimized)
The underlying models are trained on highly imbalanced data. A naive default threshold of `0.5` causes the models to predict "legitimate" for almost every case, resulting in high accuracy (e.g. 94.6%) but **0% recall**. 
In this updated version, the dashboard correctly serves metrics calculated at the **cost-optimized threshold** (e.g., ~0.27) dynamically derived from `cost_thresholds.json`. This gives an honest view of the pipeline's real-world capability (e.g. precision > 0.80 and recall > 0.70), matching our holdout and CV evaluations.

## 🔐 Environment Variables & Security

### Backend (`backend/.env`)
```env
FAGE_API_KEY=fage-demo-key-2026
NVIDIA_API_KEY=your-nvidia-nim-key-here   # Optional — SAR falls back gracefully
```

### Frontend (`frontend/.env.local`)
```env
VITE_FAGE_API_KEY=fage-demo-key-2026
```

> **Security Note**: The frontend API key is compiled into the JS bundle. For production, move auth to a cookie-based or proxy-based session.

> **Hackathon Note — Intentional Demo Backdoor**: The static API key `fage-demo-key-2026` is an intentional backdoor designed for this demonstration environment. It bypasses the JWT requirement and unconditionally grants an `admin` role with full system access. It is located in `.env.example` in plaintext so judges and evaluators can easily test the endpoints without needing to establish a full user identity.
